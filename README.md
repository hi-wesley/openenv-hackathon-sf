# DFA Agent

DFA Agent is a multi-turn OpenEnv customer-service environment. An assistant model responds to an upset customer, and the customer replies are generated by either a deterministic simulator or a local Hugging Face LLM backend.

The environment tracks four customer emotion scores on every customer reply:

- `happiness`
- `anger`
- `annoyance`
- `gratitude`

Each score is normalized to `0..1`. The assistant’s job is to move the conversation toward resolution while increasing happiness and gratitude and reducing anger and annoyance.

## What Changed

This repo now centers on three concrete support scenarios:

- `late_delivery_refund`
- `damaged_item_replacement`
- `surprise_billing_cancellation`

The assistant action is now just the next reply:

```json
{
  "message": "..."
}
```

## Why It Matters

- The benchmark is easier to understand and demo.
- The assistant is judged on customer outcomes, not abstract style matching.
- The customer side can be driven by a real LLM, so the conversation feels less scripted.

## Features

- OpenEnv `0.2.1` environment and client support
- Multi-turn customer-service episodes
- Three scenario families with train/val/test splits
- Four live customer emotion scores per turn
- Local Hugging Face customer simulator backend
- Deterministic mock simulator for offline testing
- Shaped reward pipeline plus pluggable end-of-episode scorer
- Built-in dashboard at `/dashboard` and default OpenEnv UI at `/web`
- Baseline policies, trace export, eval scripts, and GRPO training helpers

## Quick Start

```bash
python3.11 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev,train]"
bash scripts/run_local_server.sh
```

Then open:

- `http://localhost:7860/web`
- `http://localhost:7860/dashboard`

If you want a fully local run with no API calls, set `DFA_AGENT_SIMULATOR_BACKEND=mock` in [.env](/Users/wesley/Documents/Projects/openenv-hackathon-sf/.env).
If you want the local LLM-driven setup, the current default model for customer simulation, dashboard assistant generation, and training is `Qwen/Qwen2.5-0.5B-Instruct`.

## Local Development

1. Install the package in editable mode.
2. Set your simulator backend in [.env](/Users/wesley/Documents/Projects/openenv-hackathon-sf/.env).
3. Run `bash scripts/run_local_server.sh`.
4. Run `python scripts/smoke_test.py`.
5. Use `/dashboard` for the customer-service dashboard and `/web` for the default OpenEnv UI.

## Dashboard

The dashboard at `/dashboard` includes:

- full transcript
- per-turn customer emotion table
- reward breakdown by turn
- local assistant-vs-baseline control
- baseline comparison
- trace download
- story cards for the three customer-service scenarios

## Baselines

- `default_policy`
- `empathetic_policy`
- `refund_first_policy`
- `concise_policy`

Evaluation outputs include JSON traces, JSONL batches, and CSV summaries with reward, completion, parse validity, turns used, and final customer-state metadata.

## Final Scoring

The final scorer remains pluggable. The repo currently includes:

- `BaseSatisfactionScorer`
- `NoopSatisfactionScorer`
- `ConstantSatisfactionScorer`
- `EmotionBalanceScorer`

`EmotionBalanceScorer` is a lightweight built-in development scorer based on the final customer message. If you want a stronger judge, follow [docs/SCORER_PLUGIN_GUIDE.md](/Users/wesley/Documents/Projects/openenv-hackathon-sf/docs/SCORER_PLUGIN_GUIDE.md).
